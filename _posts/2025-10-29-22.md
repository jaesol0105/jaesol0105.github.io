# 2025-10-29

## Git 협업 흐름
기능브랜치 push → PR 요청 → merge

1. 개별 브랜치만들고 작업 후 커밋한다

2. 원격 저장소에 브랜치를 push한다<br>
원격저장소에 내 브랜치가 생성되고 Pull Request 버튼이 생긴다<br>
`git push origin feature/브랜치명`<br>

4. PR 버튼을 눌러서 요청한다<br>
팀원들은 깃헙 홈페이지에서 PR요청을 열람하고 리뷰한다<br>
그리고 해당 PR에 대한 merge를 승인한다<br>
PR을 요청할때는 `base(받는 쪽) dev` → `compare(보내는 쪽) feature/브랜치명`으로 설정

5. merge를 승인한다<br>
이때 방식은 3가지인데 뭐를 써야 좋은지는 잘 모르겠다<br>
GPT 피셜로 현업에서 dev로 병합할때는 squash를 많이 쓴다고한다<br>

6. merge 승인후에는 깃헙 홈페이지를 통해 원격저장소의 기능브랜치를 삭제 할 수 있다

7. 모든 기능 완성 후 최종 테스트 후 dev branch를 main branch에 최종 병합한다

<br><br>

## Merge 방식 정리
**Squash and merge** : 커밋을 1개로 정리 <br>(PR번호를 남기고 `PR로그를 통해` 기능 브랜치에서 했던 커밋내역 파악가능)<br>
**Rebase and merge** : 선형 히스토리 유지 <br>(기능 브랜치의 `커밋내역들을 다 base 쪽으로 병합해서` 반영)<br>
**Create a merge commit** : 머지 커밋 남김 <br>(머지 커밋을 남김, 히스토리 복잡해지지만 `그래프에 기능 브랜치들이 남아서` 시각화 용이)<br><br>

Squash, Rebase는 선형 히스토리가 유지되고, merge commit을 남기면 히스토리가 나누어진다.
